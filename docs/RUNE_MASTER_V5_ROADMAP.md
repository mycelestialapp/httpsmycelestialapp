# 符文大师 v5.0 落地全景指南

从理论到工程实现的完整路线图：成本、延迟、一致性、文化防火墙、国产模型选型与阶段规划。

---

## 一、成本与延迟控制

### 1.1 多级缓存

| 层级 | 实现 | TTL | 说明 |
|------|------|-----|------|
| 一级 | 前端 localStorage / IndexedDB | 7 天 | 相同问题 + 相同抽牌结果（符文+正逆位+阵法） |
| 二级 | 服务端 Redis | 24 小时 | 热门问题 + 常见抽牌组合 |
| 三级（可选） | 语义相似度 (embedding) | 可配置 | 相似度 > 0.95 且抽牌相同可复用 |

### 1.2 模型调用策略

- **80%**：规则引擎 + 轻量/本地模型（成本趋近 0）
- **20%**：高价值请求走云端顶级模型（GLM-4 / 通义等）
- 热门组合可**离线批量预生成**后写入缓存

### 1.3 成本估算（智谱 GLM-4）

- 约 0.01 元/次（500 token 级）
- 日活 1 万 × 50% 命中 LLM → 约 50 元/天，约 1500 元/月

## 二、一致性校验

### 2.1 轻量版（v1）：规则 + 黑名单

- **关键词匹配**：AI 输出不得与符文核心含义明显矛盾（如 Fehu 正位出现「损失」「破产」则降权/屏蔽）
- **规则黑名单**：禁止错误神祇对应、错误元素、错误极性
- **长度与格式**：解读控制在 100–500 字，含必要结构

违规时：**降级为规则引擎解读** 或 用规则片段替换违规句。

### 2.2 高级版（v2）：NLI 语义校验

- 使用 NLI 模型（如 BART-large-MNLI）判断「规则陈述 vs AI 句子」：矛盾 / 中立 / 蕴含
- 连续多句矛盾 → 整段不通过，可抽检或对低置信度解读复核

---

## 三、文化防火墙

### 3.1 System Prompt 强约束

- 只基于给定符文事实解读，不得编造符文含义
- 禁止：Fehu 与奥丁错误关联（应为弗雷）、Uruz 描述为水（应为土）、无关宗教混用、绝对化预测生死
- 必须引用如尼诗（若相关）、建议中须有「你的选择始终自由」类表述

### 3.2 输出扫描器

- **实体/词典**：北欧神祇、九界名与符文对应是否正确
- **黑名单词**：禁止「必然」「一定」「绝对」等（神话引述除外）
- **通用敏感词**：政治、色情、暴力等拦截

### 3.3 人工抽检与反馈

- 每日约 1% 解读人工抽检
- 用户点赞/点踩，低分进入复审与错误案例库，持续更新黑名单与规则

## 四、国产模型与提示词

### 4.1 模型选择

| 模型 | 特点 | 场景 |
|------|------|------|
| 智谱 GLM-4 | 中文诗意、指令遵循好 | 诗意解读主模型 |
| 通义千问 Max | 知识广、成本低 | 结构化解读、备选 |
| DeepSeek-V3 | 可自部署、长上下文 | 成本敏感 |
| 混元 | 安全审核强 | 强合规场景 |

### 4.2 提示词

- Few-shot + 思维链：先列出「每个符文的传统含义 → 如何组合成故事 → 如何回应用户情境」，再写解读
- 提示词版本管理，按反馈迭代

### 4.3 多模型与熔断

- 按延迟、成本、成功率做**模型路由**
- 主模型连续 3 次违规或超时 → **自动切换备选**
- 记录每次调用的模型版本便于排查

## 五、整体架构演进

| 阶段 | 周期 | 内容 |
|------|------|------|
| **MVP** | 约 1 个月 | 规则引擎完整上线；接入单模型（如 GLM-4）；规则 70% + AI 融合；本地 + Redis 缓存；轻量一致性（关键词+黑名单）；传统/灵性模式切换 |
| **优化** | 2–3 个月 | 用户历史与亲密度模型；组合解读升级；NLI 抽检；自定义阵法；多模型切换与熔断 |
| **生态** | 4–6 个月 | 社区阵法分享、点赞评论；心理热线 API；TTS 语音解读；3D 抽取与音效；可选微调专属模型 |

## 六、大师级衡量指标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 解读满意度 | ≥ 85% | 解读后「是否有帮助」等反馈 |
| 文化准确性 | ≥ 98% | 人工抽检错误率 |
| 模型幻觉率 | ≤ 3% | 抽查与规则矛盾比例 |
| 伦理警告 | 100% 覆盖 | 医疗/生死类问题必触发 |
| 平均响应时间 | ≤ 1.5 s（含缓存） | 埋点统计 |

## 七、大师级体验清单（产品侧）

- [ ] **首屏卖点**：文案明确传达「传统稳准 · 灵性更懂你」（如 rune.html 副标题）
- [ ] **节奏感**：解读先短句摘要再展开，或「灵性解读」置顶、传统四层解读紧随
- [ ] **可分享**：支持生成金句/卡片图或复制链接，便于传播
- [ ] **复访理由**：占卜档案、符文亲密度（后续阶段），让「越用越懂我」可感知
- [ ] **风格可调**：传统 / 平衡 / 灵性 三档可切换，且说明简短（传统=更稳，灵性=更懂你）

## 八、v5 首屏卖点（一句话）

- **传统模式**：九成规则脑，文化准确、不跑偏。
- **平衡模式**：规则与 AI 六四开，既稳又有灵性。
- **灵性模式**：三成规则打底，七成诗意与个性，更懂你。

## 九、代码与配置入口

- **云端（与塔罗一致）**：`supabase/functions/rune-reading/index.ts`，部署后用户无需本机开后端。前端未配置 `VITE_RUNES_API_URL` 时自动请求 Supabase 云函数。
- **云函数 Secrets（可选）**：在 Supabase Dashboard → Edge Functions → rune-reading → Secrets 中配置 `ZHIPU_API_KEY`（智谱）或 `OPENAI_API_KEY` 或 `DEEPSEEK_API_KEY` 可启用灵性融合；不配置则仅返回规则解读。
- **自建后端**：`backend/main.py`、`runes_data.py` 等；配置 `VITE_RUNES_API_URL=http://localhost:8000` 时前端请求自建后端（本地联调用）。
- **前端**：`RunesPanel` 优先用 Supabase 云函数（与塔罗/雷诺曼一致），有 `VITE_RUNES_API_URL` 时改用自建 API。
