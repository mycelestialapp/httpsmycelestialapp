# 星圖板塊 · 西方占卜內容生成邏輯

本文件**參照八字板塊的制定方式**（免費版／付費版兩套表達、結構分層、用詞規範），定義**星圖板塊內西方占卜工具**的內容生成邏輯，供前後端對齊實作。  
八字板塊已定好，此處僅制定星圖板塊的西方占卜（占星、塔羅、雷諾曼、數字命理等）的生成規則。

---

## 一、制定方式對齊（參照八字）

八字板塊的制定方式要點：

- **兩套表達**：免費版（簡短、固定版位）、付費版（詳細、不限字數、可含更多章節）。
- **結構分層**：按「一、二、三…」或「【區塊】」分層，付費版多「【詳解】」類擴展。
- **用詞規範**：付費版避免「具體建議」「具體指引」「具體宜忌」等萬能話術；改為依命盤/牌局**推導**出的描述與傾向。
- **視覺區分**：付費版主標題使用金色等強調樣式；免費版可保持常規字色。
- **數據驅動**：內容由後端計算結果（如尊貴度、相位、牌位）驅動，而非固定文案隨機拼貼。

星圖板塊西方占卜的生成邏輯按上述同一套思路制定：**免費版 / 付費版兩套、結構清晰、用詞規範、與 Logic Kernel 輸出對齊**。

---

## 二、適用範圍與數據分級

- **適用範圍**：星圖板塊內所有「西方占卜」相關內容，包括：
  - 占星（本命盤、過境、簡易合盤）；
  - 塔羅（單牌、三張、凱爾特十字等）；
  - 雷諾曼（單牌、三張、大桌）；
  - 數字命理（生命靈數、流年數等）；
  - 由 Orchestrator 融合後的綜合報告（若存在）。
- **數據分級**（與現有《西方占卜-星圖板塊內容生成方案》一致）：
  - **L1 僅日期**：太陽星座 → 靈魂原型、金句、關鍵詞等。
  - **L2 日期+時間**：太陽+月亮 → 在 L1 基礎上增加情緒／內在維度。
  - **L3 日期+時間+地點**：完整本命盤（上升、宮位、行星落宮、尊貴度、相位）→ 可產出完整四層報告。

生成邏輯中，**免費版可僅使用 L1（或 L2）**；**付費版應使用 L2 或 L3**，並與 Kernel 的 `AstrologyOutput` / `TarotOutput` / `LenormandOutput` 等結構對齊。

## 三、占星（Astrology）生成邏輯

### 3.1 免費版

- **內容範圍**：
  - 若僅有日期（L1）：**太陽星座**一句話性格 + 一句能量關鍵詞 + 一句金句（從 12 星座預設表讀取）。
  - 若有日期+時間（L2）：在 L1 基礎上 + **月亮星座**一句話（情緒／內在）+ 一句「太陽×月亮」組合短評。
- **結構**：
  - 無需「一、二、三」多層，單一區塊即可，總字數建議 **150～300 字**。
- **用詞**：
  - 可為通用星座描述，但**不得出現**「具體建議」「具體指引」「具體宜忌」；可改為「本命傾向」「能量特質」等中性表述。
- **數據來源**：
  - 輸入：`birth_iso8601`（或僅 `birth_date`）、可選 `lat_lng` / `tz_offset`。
  - 輸出：僅使用 `core_points.Sun`（及可選 `core_points.Moon`），或僅用太陽星座計算結果（無需完整星盤）。

### 3.2 付費版

- **內容範圍**：
  - **第一層**：本命結構分析（核心：太陽落宮、上升、月亮與金星等關鍵點），含架構小結。
  - **第二層**：尊貴度明細（入庙/旺/陷/落及對應 Dignity Score），附 Dignity Sum 與一句解讀。
  - **第三層**：能量總分計算（Energy Total 公式與代入結果）、重要相位列表及一句解讀。
  - **第四層**：事業/財運（或用戶問的主題）深度結論，**必須依尊貴度與相位推導**，禁止萬能模板；若存在太陽與土星合相，需明確寫出「克制與長遠布局」對財運的具體影響。
- **結構**：
  - 嚴格按「第一層」「第二層」「第三層」「第四層」分層；可加「唯一性總結」與「置信度標註」。
  - 字數**不限**，以說清邏輯鏈為準。
- **用詞**：
  - **禁止**「具體建議」「具體指引」「具體宜忌」；改為「從命盤看…」「依相位…」「尊貴度顯示…」等有依據的表述。
  - 語氣：冷靜、客觀、具職業洞察力，類似頂級私人顧問口吻。
- **數據來源**：
  - 輸入：符合 `AstrologyInput`（birth_iso8601, lat_lng, tz_offset）。
  - 輸出：對齊 `AstrologyOutput`（core_points, planets, aspects, dignity_sum, energy_total, meta）；文案由上述字段生成，不得脫離數據編造。
- **UI**：
  - 主標題（第一層、第二層、第三層、第四層）使用**金色**（與八字付費版一致）。

## 四、塔羅（Tarot）生成邏輯

### 4.1 免費版

- **內容範圍**：
  - 單次抽牌或三張牌：每張牌**一句標題 + 一句釋義**（從牌義表讀取），再給**一句整體主題**（可依牌陣位置權重加權）。
  - 總字數建議 **120～250 字**。
- **結構**：
  - 單一區塊：牌名 + 簡短釋義 + 整體主題。
- **用詞**：
  - 禁止「具體建議」「具體指引」「具體宜忌」；可為「牌義傾向」「當下能量」等中性表述。
- **數據來源**：
  - 輸入：`spread_type`、可選 `question`；RNG 用固定或簡化 seed。
  - 輸出：對齊 `TarotOutput` 的 `slots`（card_id, upright, element）、`overall_theme`、`confidence`；免費版可僅輸出精簡版 slots + 一句 overall_theme。

### 4.2 付費版

- **內容範圍**：
  - 每張牌：牌名、正逆位、元素、**位置意義**（如過去/現在/未來）、**與其他牌的元素互動**（依 ELEMENT_WEIGHT_MATRIX）、一句該牌在該位置的解讀。
  - 整體：**綜合主題** + **2～3 條有依據的傾向**（由牌位與元素互動推導），不寫萬能建議。
- **結構**：
  - 區塊一：牌陣與各牌解讀（逐張）。
  - 區塊二：綜合主題與傾向（依邏輯鏈寫）。
  - 可選：置信度、Seed 記錄說明（不洩露 seed 值）。
- **用詞**：
  - 禁止「具體建議」「具體指引」「具體宜忌」；改為「此牌在此位顯示…」「元素互動顯示…」。
- **數據來源**：
  - 輸入：符合 `TarotInput`（spread_type, question, 可選 seed_override）。
  - 輸出：完整對齊 `TarotOutput`（seed_used, spread_type, slots 含 position_weight / element_interaction_weight / slot_score, overall_theme, confidence）。
- **UI**：
  - 主標題使用**金色**。

## 五、雷諾曼（Lenormand）生成邏輯

### 5.1 免費版

- **內容範圍**：
  - 單張或三張：每張**牌名 + 宮位/位置** + **一句簡義**（從牌義表讀取）；整體一句**組合主題**（若多張）。
  - 總字數建議 **100～220 字**。
- **結構**：
  - 單一區塊。
- **用詞**：
  - 禁止「具體建議」「具體指引」「具體宜忌」。
- **數據來源**：
  - 輸入：`layout`、`cards`（牌+宮位）；可無需 Distance_Rule / Knighting 計算。
  - 輸出：可為精簡版 `LenormandOutput`（syntactic_interpretation.core_phrase + confidence），或僅 core_phrase + 一句 evidence。

### 5.2 付費版

- **內容範圍**：
  - **空間邏輯**：Distance_Rule（與 Significator 或關鍵牌的距離）、Knighting（跳馬位關聯）、可選 Mirroring/Diagonals，每項一兩句說明。
  - **組合語法**：依 COMBINATORY_SYNTAX（名詞+形容詞+宮位語境）給出**一句核心短語** + **2～3 條證據句**（evidence）。
  - 結論：由上述邏輯推導，**禁止萬能模板**。
- **結構**：
  - 區塊一：空間與距離（含 distance_bucket、knighting_links 等）。
  - 區塊二：組合解讀（core_phrase + evidence）。
  - 可選：置信度。
- **用詞**：
  - 禁止「具體建議」「具體指引」「具體宜忌」；改為「此牌在此宮位顯示…」「距離與跳馬顯示…」。
- **數據來源**：
  - 輸入：符合 `LenormandInput`（layout, cards, 可選 significator）。
  - 輸出：完整對齊 `LenormandOutput`（spatial.metrics, syntactic_interpretation.core_phrase / evidence / confidence）。
- **UI**：
  - 主標題使用**金色**。

## 六、數字命理（Numerology）生成邏輯

### 6.1 免費版

- **內容範圍**：
  - 生命靈數（Life Path）一數 + **一句定義**（從預設表讀取）+ 一句「本命傾向」類短評。
  - 總字數建議 **80～150 字**。
- **用詞**：
  - 禁止「具體建議」「具體指引」「具體宜忌」。
- **數據來源**：
  - 輸入：`birth_date`（或 + `full_name` 僅取 Life Path）。
  - 輸出：對齊 `NumerologyOutput` 的 `life_path`（value, is_master, karmic_debts）。

### 6.2 付費版

- **內容範圍**：
  - 生命靈數 + 表達數（Expression）+ 靈魂數（Soul Urge）+ 人格數（Personality）；每項一句定義 + 一句與命盤/主題的結合（若與占星等融合）。
  - 若有 Karmic Debt，單獨一句說明，不寫「具體宜忌」。
- **結構**：
  - 按「生命靈數」「表達數」「靈魂數」「人格數」分小節；可選「與星圖結合」一節。
- **用詞**：
  - 禁止「具體建議」「具體指引」「具體宜忌」。
- **數據來源**：
  - 輸入：符合 `NumerologyInput`（birth_date, full_name）。
  - 輸出：完整對齊 `NumerologyOutput`（life_path, expression, soul_urge, personality）。
- **UI**：
  - 主標題使用**金色**。

## 七、Orchestrator 融合報告（若啟用）

- **免費版**：不出融合報告，或僅出一句「綜合主題」+ 置信度。
- **付費版**：
  - 依 `OrchestratorOutput`（modules_used, module_summaries, fused_theme, fused_advice, confidence_score）生成一段融合報告；
  - 內容須註明由哪些模塊組成、權重如何，結論不得脫離各模塊輸出；
  - 主標題使用**金色**。

---

## 八、統一用詞與禁用詞（兩版皆須遵守）

- **付費版一律禁止**：
  - 「具體建議」「具體指引」「具體宜忌」及同義表述（如「具體的開運方法」「具體的忌諱事項」）。
  - 改為：依命盤/牌局/數字的**推導性描述**與**傾向**（如「從命盤看…」「此牌位顯示…」「尊貴度顯示…」）。
- **免費版**：
  - 建議同樣避免上述三類詞，以保持與付費版一致的專業感；若需「建議」類表述，可用「本命傾向」「能量特質」「可多留意…」等中性語。
- **語氣**：
  - 冷靜、客觀、具職業洞察力；不保證、不絕對化（避免「一定會」「絕對不要」）。

## 九、與 Logic Kernel 的對應關係

| 模塊       | 免費版生成依據              | 付費版生成依據                    |
|------------|-----------------------------|-----------------------------------|
| 占星       | core_points.Sun（及可選 Moon） | 完整 AstrologyOutput（四層+公式） |
| 塔羅       | TarotOutput 精簡（slots + overall_theme） | 完整 TarotOutput（含權重與互動）  |
| 雷諾曼     | LenormandOutput 精簡（core_phrase） | 完整 LenormandOutput（spatial + syntactic） |
| 數字命理   | NumerologyOutput.life_path | 完整 NumerologyOutput            |
| 融合報告   | 可無或一句                  | OrchestratorOutput               |

生成邏輯實作時，應從 Kernel 計算結果（或 mock）讀取上述字段，再按「免費版／付費版」規則組裝文案，**不脫離 Kernel 輸出**編造內容。

---

## 十、實作建議

1. **新增模塊**：可為星圖西方占卜單獨建立「內容生成層」（例如 `westernDivinationReading.ts` 或按工具拆成 `astrologyReading.ts`、`tarotReading.ts` 等），與現有 `baziReading.ts` 平行；**不修改八字邏輯**。
2. **入口參數**：與八字一致，建議傳入 `level: 'short' | 'long'` 或 `variant: 'free' | 'paid'`；若用 `variant`，需在調用 Kernel 時區分請求的數據深度（L1/L2/L3）。
3. **UI 層**：星圖結果展示組件根據 `variant` 或 `level` 決定是否使用金色主標、是否展開「詳解」區塊，與八字付費版主標題樣式保持一致即可。

以上為星圖板塊西方占卜的生成邏輯制定，與八字板塊的制定方式對齊，便於後續實現與維護。
